# Anti-Slop System (反套路系统)

LLM 训练数据里的高频模式 = "slop"。创意生成最大的失败不是"想不出"，是"想出来的都是训练集最常见的 20 种组合"。这一文件是 22 方法**之前**必须先过的筛子。

## What is slop?

Slop = 输出一种把名词替换成同义词就能变出下一个答案的模式。

**最常见的 slop 形态**（用方法 #5 Lateral Provocations 的命名）：
- "AI 驱动的 X"（AI-powered / AI-driven / 用 AI 重新定义 / 让 AI 改变）
- "Uber for Z"（用一个真实公司 + 一个领域）
- 习惯追踪器（habit tracker / mood tracker / time tracker / X tracker）
- 健身 App（workout planner / meal planner / calorie / hydration）
- 两个时髦名词的随意拼接（"Web3 + 健身"、"AI + 冥想"）
- Y Combinator 风格的名字（两个音节、去掉元音、.ai 域名、.so / .xyz）
- "为你 X 的一切需求"
- "重新定义 [X]"
- "终极 [X] 工具"
- 模板化的 App 一句话说明：「[X] 是一款 [Y] 工具，帮助用户 [Z]，让用户 [W]」

**判断方法**：把每个 idea 的名词换成另一个高频名词，看 idea 是否基本没变化。如果是 → slop。

## Layer 1: 拒绝前 3 个（基础）

无论用什么方法，**先默认生成 3 个，不输出，丢掉**。再生成 3 个输出。

- 前 3 个反映 LLM 基线分布（argmax over training priors）
- 后 3 个才能开始偏离基线
- 在高套路领域（"AI 创业"、"效率工具"、"副业"、"学习"）→ **拒绝前 5 个**

## Layer 2: 强迫具体化

每个 idea 必须包含**至少 3 个**具体锚点：
- **真实人名**（"Wei, 32, 上海做外贸"）
- **真实地名**（"上海徐家汇地铁站出口"）
- **真实材料/介质**（"用 200 行 Python + Click"）
- **具体机制**（"在终端里画出"）
- **数字**（"3 个按键"、"47 毫秒"、"200ms 内"）

**反例**（slop）：
- ❌ "一个连接创作者和粉丝的平台"
- ❌ "智能化的健身助手"
- ❌ "用 AI 提升写作效率"

**正例**（具体化）：
- ✅ "一个 200 行的 CLI 工具，在终端里画出你过去 30 天没打开的浏览器标签页，按语义距离排序"
- ✅ "Wei 在上海徐家汇地铁站出口 3 分钟的步行中，可以通过 1 个机械键盘快捷键，给当周联系过的 47 个外贸客户发不同语言的跟进消息"
- ✅ "一本 96 页的 A6 开本口袋书，封面是牛皮纸，内容是用户过去 12 个月 GitHub commit 画的星座图"

### Layer 2.5: 锚点多样性 (v2.1 新增)

为防止 LLM 全部输出"工程细节"型锚点（端口号、文件路径、CLI 命令），每个 idea 的 ≥3 个锚点应包含至少：

- **1 个"用户/场景"锚点**（具体人名 / 城市 / 场景 / 时间 / 情绪 / 当事人观点）
- **1 个"机制/工程"锚点**（具体技术 / 数字 / 接口 / 时间 / 文件名 / 命令）
- **1 个"反常/边界"锚点**（违反 best practice / 不寻常假设 / 边缘 case）

LLM 默认偏向"工程细节"型锚点（如端口号、文件路径、CLI 命令），忽视"用户/场景"型锚点（人名、城市、场景、情绪）。这两类必须配对使用，否则 idea 容易变成"工程师自嗨"。

## Layer 3: 怪诞预算 (Grotesque Budget)

每个 idea 里**至少一个需要解释的元素**。

- 全部常规 → 平庸
- 全部怪诞 → 不可执行
- 混合：80% 可执行 + 20% 需要解释

**怪诞预算的实现**：
- 至少 1 个 idea 在今天就能动手做（非显而易见，但接地气）
- 至少 1 个 idea 包含一个违反 best practice 的选择（用 5-Why 分析过为什么 ok）
- 至少 1 个 idea 让你或用户"想跟朋友解释它是什么" — 解释成本高 = 高怪异度

**v2.1 警告：装饰性怪诞不算怪诞**。纪念碑 / random emoji / 5 秒环境音归档这类"调味料"型怪诞看似有趣，但不解决核心问题。怪诞必须**提升核心价值**（如"双皮肤自动切换"提升渠道-调性匹配、"双向只追加 vault 索引"提升双向可追溯），不只是"看起来怪"。

## Layer 4: Slop 体检（生成后自检）

输出 3-5 个 idea 后，跑这个 checklist：

- [ ] 每个 idea 换主语（人/公司/领域）后，是否基本等价？→ 如果 yes，是 slop
- [ ] 每个 idea 都能在 YC 申请书的 1 句话里说清吗？→ 如果 yes，可能太套路
- [ ] 每个 idea 都能在 Indie Hackers 上找到至少 3 个现成产品？→ 如果 yes，没差异化
- [ ] 把 idea 念给一个完全外行的人，他能复述出来吗？→ 如果 yes，可能太通顺
- [ ] 你看到这个 idea 时感到"惊喜"吗？→ 如果 5 个 idea 都没有，重来

## Layer 5: 引用与数据真实 (v2.1 新增, 修 Bug #2)

LLM 在"举例说明"时倾向于编造具体产品/数据/案例作为论据。这是"幻觉式引用"slop 的一种。

**规则**：

- **引用具体产品/公司/案例时，必须能说出至少 1 个具体事实**（年份、创始人、URL、关键事件）。说不出 → 删掉。
- **引用统计数据时，必须用模糊语或注明来源**。"很多"、"一些"、"据 [来源]" 优于 "70%"。
- **不要编造看似权威的引语**。如果想引用某人的话，要么给出来源链接，要么用 "据我理解" / "概念上"。
- **如果不确定，就说"我不确定"或"假设"**。主公更愿意听"我可能错了" 也不愿意听"绝对"。

**反例**（幻觉式引用）:
- ❌ "Idle Slayer 月流水过百万" — 不知道是不是真的，但听起来权威
- ❌ "《大多数》手游丑出了梗流量反而涨" — 不知道是不是真的
- ❌ "CrazyGames 暗黑/写实点击率比卡通高 30-40%" — 不知道是不是真的

**正例**（诚实表述）:
- ✅ "Idle Slayer 这类休闲点击游戏在 Steam 长期存在，验证了'挂机玩家可能不挑美术'的假设（具体流水我没数据）"
- ✅ "可能存在'丑即 feature'的手游案例，但我举不出具体名字，建议主公回忆一下"
- ✅ "暗调点击率高的说法我听到过但没数据支持，假设如此"

**主公偏好**：诚实标注"假设"/"我不确定" > 编造权威细节。主公会在看到"我可能错了"时信任 agent，看到"绝对"时质疑 agent。

## Layer 6: 推送前链式安全检查 (v2.2 新增)

如果用户原话含"推"/"GitHub"/"开源"/"PR"等触发词（详见 `overrides.md` Rule 9），在 anti-slop 5 层跑完之后，**自动调 `gh-push-safety-check` skill** 验证要推的代码路径没敏感信息。

**这是 anti-slop 体系的"外部护栏"**：Layer 1-5 是文本层过滤（idea 不能 slop），Layer 6 是物理层过滤（idea 落地的代码不能 leak）。

**不替代 Layer 1-5**：Layer 6 只检查 *即将推送的代码*，不检查 *输出的 idea 文本*。两者正交。

**通过条件**: `gh-push-safety-check --strict` 返回 exit 0 (PASS)。

**失败处理**:
- 顶部加 ⚠️ `BLOCK PUSH: <N> critical findings` 红色警告
- idea 仍输出（ideation 与 execution 解耦）
- 列出 top 5 findings 文件:行号 + 类别
- 告诉用户去 `~/.hermes/skills/gh-push-safety-check/SKILL.md` 看 6 类说明

**主公原话（06-25）**: "记住像这样推上去之前检查有没有敏感信息" — 这条 Layer 6 是直接落实这个偏好的机制化版本。

## 实现说明

Anti-slop 不是一个单独方法。它是**所有方法共用的过滤器**。在用 22 任一方法生成 idea 后，必须跑 Layer 1-3 的检查。

## 反 slop 失败信号

如果用户连续 2 轮反馈"还是太常规"或"AI 味"或"无聊"：
- 切换到方法 #5 Lateral Provocations
- 启动方法 #10 Pataphysics
- 加 Layer 4 的 5 条自检
- 至少再生成 5 个 — **其中必须有 1 个完全荒诞但有逻辑链**
